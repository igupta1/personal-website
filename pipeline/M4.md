# Plan — Milestone 4: `enrichment.py`

## Context

Per `pipeline/ROADMAP.md` M4: given a `Lead`, fill in `industry`, `headcount`,
`country` via two passes: (a) deterministic compute, (b) LLM classification
through `msp_pipeline.llm`. Per the M1 redline the schema only has those three
enrichment columns (state/city/founded_year/linkedin_url/description were
cut), so M4 mirrors that scope.

## Verified state

- `pipeline/msp_pipeline/` has `models.py`, `db.py`, `sources/`, `llm.py`. No
  `enrichment.py` yet.
- `db.update_lead`'s `_UPDATABLE_FIELDS` allowlist already covers `industry`,
  `headcount`, `country` — no schema change needed for the columns we write.
- **Small `db.py` addition required:** `delete_lead(conn, lead_id)` — needed
  by the country-filter step (per redline #3). One-liner.
- **`models.py` changes required:**
  - Drop `SignalType.HEADCOUNT_THRESHOLD_CROSSED` and
    `SignalType.HEADCOUNT_GROWTH_RAPID` per redline #4. Neither is written
    anywhere yet, so removal is safe; M5 scoring will not reference them.
  - Add `SignalType.LOCATION_CAPTURED = "location_captured"` to back the
    city/state capture (locked option 2 below). Written by `enrich()` with
    `source = SourceName.COMPUTED` and `payload = {"city": ..., "state": ...}`.
  - Add `SignalType.ENRICHMENT_RUN = "enrichment_run"` to mark the last
    enrichment for signal-aware re-enrichment (locked decision #5 below).
    Written by `enrich()` with `source = SourceName.COMPUTED` and empty
    `payload = {}`. Acts as a timestamp marker only.
    `SourceName.COMPUTED` was kept in M3 cleanup specifically for these
    enrichment-emitted signals.
- `llm.call_openai(prompt, response_model=...)` returns a Pydantic instance via
  forced JSON schema (gpt-4o-mini default).
- `llm.call_gemini(prompt)` returns plain text **with Google Search grounding**
  enabled — this is the right tool for headcount + location lookup since
  ungrounded LLMs hallucinate sizes for unknown companies.
  `call_gemini(prompt, response_model=...)` drops grounding (SDK constraint),
  so the lookup uses the text path and parses fields out via regex.
- M2 source modules emit signals but never headcount/industry/country.
  Enrichment is the only place those fields are populated in v1.

## Module shape — `msp_pipeline/enrichment.py`

```python
from __future__ import annotations
import logging, re, sqlite3
from enum import Enum
from pydantic import BaseModel
from msp_pipeline import db, llm
from msp_pipeline.models import Lead

log = logging.getLogger(__name__)

# --- Industry vocabulary (closed enum per redline #1) ----------------------

class Industry(str, Enum):
    SOFTWARE_SAAS         = "software_saas"
    FINTECH               = "fintech"
    HEALTHCARE            = "healthcare"
    ECOMMERCE_RETAIL      = "ecommerce_retail"
    MANUFACTURING         = "manufacturing"
    LOGISTICS_TRANSPORT   = "logistics_transport"
    REAL_ESTATE           = "real_estate"
    LEGAL_PROFESSIONAL    = "legal_professional"
    EDUCATION             = "education"
    MEDIA_ENTERTAINMENT   = "media_entertainment"
    HOSPITALITY_FOOD      = "hospitality_food"
    GOVERNMENT_NONPROFIT  = "government_nonprofit"
    CONSTRUCTION          = "construction"
    OTHER                 = "other"

# 14 tags, including OTHER as the LLM's escape valve. Tuned for MSP targeting:
# healthcare/legal → strong it_msp targets, fintech/healthcare → mssp,
# software_saas → cloud, etc. M5 scoring keys directly off these strings.

# --- Pure helpers (deterministic; trivially testable) ----------------------

def compute_band(headcount: int | None) -> str | None:
    """Headcount band on read. Not stored in v1 (per M1 redline)."""
    # None → None; <10 → "1-10"; <50 → "11-50"; <200 → "51-200";
    # <1000 → "201-1000"; <5000 → "1001-5000"; else → "5000+".

def _round_to_10(n: int) -> int:
    """Round to nearest 10 per redline #2. round(n, -1) handles banker's
    rounding cleanly; for our use case the half-up vs banker distinction
    doesn't matter."""
    return int(round(n, -1))

# --- Combined LLM lookup (one Gemini grounded call per lead) ---------------

# Single grounded Gemini call returns headcount + country (+ optional
# location capture per redline #3). Combining cuts cost vs separate calls.

_LOOKUP_PROMPT = """\
Look up the company "{name}" on the web. Reply with EXACTLY these lines and
nothing else (use "unknown" when you cannot determine a value):

HEADCOUNT: <integer employee count or "unknown">
CITY: <city name or "unknown">
STATE: <2-letter US state code (CA, NY, ...) or "unknown">
COUNTRY: <2-letter ISO country code (US, GB, CA, ...) or "unknown">
"""

_FIELD_RE = {
    "headcount": re.compile(r"^HEADCOUNT:\s*(.+)$", re.MULTILINE | re.IGNORECASE),
    "city":      re.compile(r"^CITY:\s*(.+)$",      re.MULTILINE | re.IGNORECASE),
    "state":     re.compile(r"^STATE:\s*(.+)$",     re.MULTILINE | re.IGNORECASE),
    "country":   re.compile(r"^COUNTRY:\s*(.+)$",   re.MULTILINE | re.IGNORECASE),
}

class _Lookup(BaseModel):
    headcount: int | None
    city:      str | None
    state:     str | None
    country:   str | None  # ISO-2; "US" for the keep filter

def lookup_company(lead: Lead) -> _Lookup:
    """One Gemini grounded text call → parsed structure. Headcount rounded
    to nearest 10."""

# --- Industry classification (OpenAI structured) ---------------------------

class _IndustryOut(BaseModel):
    industry: Industry  # enum-constrained per redline #1

def classify_industry(lead: Lead) -> Industry:
    """Short structured prompt with the 14-tag enum. OpenAI strict-mode
    JSON schema enforces the value at the API level; OTHER is the
    fallback."""

# --- Orchestrator ----------------------------------------------------------

def enrich(conn: sqlite3.Connection, lead: Lead, *, force: bool = False) -> bool:
    """Returns True if the lead was kept (and possibly updated), False if
    it was deleted (non-US per redline #3).

    Signal-aware skip (per redline #5): skip the LLM passes when
      lead.industry is not None
      AND no source signal arrived after the most recent ENRICHMENT_RUN
      AND not force
    Otherwise, run the full enrichment pipeline.

    Pipeline:
      1. lookup_company(lead) — 1 Gemini grounded call
      2. if country != "US" OR headcount > 250: db.delete_lead; return False
      3. if city or state present: db.append_signal(LOCATION_CAPTURED,
         source=COMPUTED, payload={"city": ..., "state": ...})
      4. classify_industry(lead) — 1 OpenAI structured call
      5. db.update_lead(industry=..., headcount=..., country="US")
      6. db.append_signal(ENRICHMENT_RUN, source=COMPUTED, payload={})
      7. return True
    """
```

Module-level constants:
- `_LOOKUP_PROMPT` — the structured-text Gemini prompt above.
- `_INDUSTRY_PROMPT` — "Given the company name and recent signal types
  below, classify the company into one of these industries: {tags}. Reply
  with just the tag." (followed by `name=...` and signal summary).
- `_summarize_signals(lead)` returns a short string like
  `"jobs (security, cloud_devops); funding"` so the LLM has signal context
  without dumping payloads.

## City/state capture (locked — option 2)

Captured in `_LOOKUP_PROMPT`, persisted as a `Signal` of type
`LOCATION_CAPTURED` (source `COMPUTED`) on the lead's `signals` JSON. No
schema change. Skipped entirely if both `city` and `state` come back
`None` from the lookup — no point appending an empty payload.

Future geography filter: ALTER TABLE adds a `state` column, then a one-off
backfill script reads each lead's most recent `LOCATION_CAPTURED` signal
and copies `payload["state"]` into the new column. No re-query of the LLM
needed.

## Locked decisions (post-redline)

1. **Industry vocabulary — closed enum of 14 tags** (`Industry` in
   `enrichment.py`). Constrained at the OpenAI API layer via
   `response_format=_IndustryOut` where `industry: Industry`. M5 scoring
   keys directly off these strings, so changes to the enum after M5 lands
   require a coordinated update.
2. **Headcount — Gemini grounded text, rounded to nearest 10.** Captured
   via the combined `lookup_company` call (one Gemini call total, not
   two). Rounding is half-even via Python `round(n, -1)`.
3. **Country — LLM-verified per lead, non-US leads deleted.** The combined
   Gemini lookup returns ISO-2 country code; if not `"US"`, `enrich()`
   calls `db.delete_lead(lead.id)` and returns `False`. The deterministic
   "all signals from US sources" heuristic is dropped — LLM verification
   is more robust now that we're paying for one Gemini call per lead
   anyway.
   - **Geography filter (city/state) — locked option 2:** captured by the
     same Gemini call, persisted as a `LOCATION_CAPTURED` signal on the
     lead's `signals` JSON. No schema change; future geography filter
     adds a `state` column + backfills from existing signals (no LLM
     re-query needed).
4. **Computed signals — fully removed.** `HEADCOUNT_THRESHOLD_CROSSED` and
   `HEADCOUNT_GROWTH_RAPID` are deleted from `SignalType` in `models.py`.
   No code path in v1 wants temporal headcount diffing. If we ever add it,
   we add the enum value back at that time. `SourceName.COMPUTED` stays
   (cheap to keep, future hatch).
5. **Re-enrichment — signal-aware skip.** Skip when `lead.industry` is set
   AND no source signal has arrived after the lead's most recent
   `ENRICHMENT_RUN` signal. Re-fire when a new source signal arrives
   (Option B from chat — timestamp via `ENRICHMENT_RUN` marker, no schema
   change). `force=True` overrides the skip in either direction (e.g.
   prompt or enum changes). The "newer than the last enrichment" check
   uses `Signal.captured_at`; ties (`>=`) are treated as not-newer to
   avoid re-firing on the just-stamped marker. Cost: bounded by signal
   arrival rate (typically ≤ once per week per lead at our cron volume).
8. **SMB cap — delete leads with `headcount > 250`** (added in M5 redline).
   Strict inequality; exactly 250 is kept. Unknown headcount (Gemini
   returned "unknown") is kept — those are usually small private
   companies that fit our SMB target. The check runs alongside the non-US
   filter: if `country != "US"` OR `headcount > 250`, `db.delete_lead`
   and return `False`. Industry classification still runs for kept leads
   so the UI can filter on it.
6. **Prompts — module-level constants in `enrichment.py`.** Same pattern
   as M2's keyword lists. Confirmed.
7. **Commit granularity — single commit `add enrichment.py` (plus the
   small `db.delete_lead` and `models.py` enum trim).** Confirmed.

  `100000 → "5000+"`.

  return a `HEADCOUNT: 1247\nCITY: Boston\nSTATE: MA\nCOUNTRY: US` block;

  `_IndustryOut(industry=Industry.FINTECH)`; verify the function returns

- **`test_enrich_persists_headcount_rounding`** — mock `call_gemini`

## Out of scope (per `pipeline/ROADMAP.md` and the redlines above)

- Computed signals (`HEADCOUNT_THRESHOLD_CROSSED`, `HEADCOUNT_GROWTH_RAPID`)
  — **fully removed** per redline #4 (not just deferred). Add back the enum
  values and emission logic if/when temporal headcount diffing actually
  matters.
- Geography filter (state-level) — data is captured into
  `LOCATION_CAPTURED` payloads but no column or filter UI is built in M4.
  A future milestone adds an `ALTER TABLE leads ADD COLUMN state TEXT`,
  backfills from each lead's most recent `LOCATION_CAPTURED` signal, and
  exposes a state filter on the React pages — no LLM re-query needed.
- Per-niche scoring — M5. M5 will key off `Industry` enum values.
- LLM-assisted company-name cleanup for raw RSS funding titles — would
  require touching `name_key` post-upsert, which breaks dedup. Defer to a
  M2 follow-up or pre-upsert middleware later.
- Headcount-band column in the DB — M1 redline keeps it as a read-time
  computation only.
- `founded_year`, `linkedin_url`, `description` columns — M1 redline cut
  them. Add via `ALTER TABLE` only if a later milestone needs them.
- Real headcount data APIs (Apollo, Crunchbase, etc.) — M4 uses LLM-grounded
  estimates rounded to nearest 10; a paid API can replace the headcount
  field of `lookup_company` later without changing the public surface.
