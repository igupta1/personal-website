# Plan — Milestone 10: GitHub Actions cron

## Context

Per `pipeline/ROADMAP.md` M10: schedule the pipeline to run nightly via
GitHub Actions. The committed `pipeline/data/leads.db` carries enrichment
state across runs (per `pipeline/CLAUDE.md`: "`data/leads.db` is committed
to git so the lead set persists across runs"). Each cron run mutates that
DB, then commits it back so tomorrow's run picks up where today's left
off — incremental, not full-rescrape-each-time.

This is the last milestone. After M10, the system runs autonomously.

## Module shape — `.github/workflows/daily-leads.yml`

```yaml
name: Daily lead refresh

on:
  schedule:
    - cron: "0 6 * * *"   # 06:00 UTC daily (~11pm PT, 2am ET)
  workflow_dispatch:       # manual trigger via Actions tab

permissions:
  contents: write          # needed to push the DB commit back

concurrency:
  group: daily-leads
  cancel-in-progress: false

jobs:
  refresh:
    runs-on: ubuntu-latest
    timeout-minutes: 30
    steps:
      - name: Checkout
        uses: actions/checkout@v4
        with:
          fetch-depth: 1
          token: ${{ secrets.GITHUB_TOKEN }}

      - name: Set up Python
        uses: actions/setup-python@v5
        with:
          python-version: "3.13"
          cache: "pip"
          cache-dependency-path: pipeline/pyproject.toml

      - name: Install pipeline
        working-directory: pipeline
        run: |
          python -m pip install --upgrade pip
          pip install -e .

      - name: Run daily pipeline
        working-directory: pipeline
        env:
          OPENAI_API_KEY:       ${{ secrets.OPENAI_API_KEY }}
          GEMINI_API_KEY:       ${{ secrets.GEMINI_API_KEY }}
          ADZUNA_APP_ID:        ${{ secrets.ADZUNA_APP_ID }}
          ADZUNA_APP_KEY:       ${{ secrets.ADZUNA_APP_KEY }}
          LEADS_UPLOAD_URL:     https://www.ishaangpta.com/api/upload-leads
          LEADS_UPLOAD_API_KEY: ${{ secrets.LEADS_UPLOAD_API_KEY }}
        run: python -m msp_pipeline.daily_run --upload

      - name: Commit updated DB + JSON
        if: success()
        run: |
          git config user.name  "leads-cron[bot]"
          git config user.email "leads-cron[bot]@users.noreply.github.com"
          if [[ -n "$(git status --porcelain pipeline/data/leads.db pipeline/data/leads.json)" ]]; then
            git add pipeline/data/leads.db pipeline/data/leads.json
            git commit -m "cron: nightly lead refresh ($(date -u +%Y-%m-%d))"
            git push
          else
            echo "No changes to commit"
          fi
```

## Locked-pending-redline decisions

1. **Schedule — `0 6 * * *`** (06:00 UTC daily). Lands ~11pm PT / 2am ET
   — prospects' inboxes are quiet, latest job postings/funding/breach
   data is captured. Easy to change later.
2. **Concurrency — `cancel-in-progress: false`.** If a manual trigger
   fires while a scheduled run is mid-flight, queue the manual run
   instead of cancelling. Avoids partial-state DB commits. The job
   has a 30-min timeout safety net.
3. **Commit only on `success()`.** If the pipeline or upload fails, the
   DB is *not* committed back. The next run re-runs against the
   previously committed DB. Cost: re-enriching whatever was enriched
   in the failed run (a handful of LLM calls, well under a cent).
   Benefit: no half-finished state ever lands in git history.
4. **Commit BOTH `leads.db` AND `leads.json`.** The JSON is also stored
   on Vercel Blob (the source of truth for the React UI), but
   committing it to git gives a free historical record — useful for
   "what did the lead set look like 30 days ago" debugging. The DB is
   the load-bearing artifact.
5. **`LEADS_UPLOAD_URL` hardcoded in the YAML**, not a secret/variable.
   The URL is public (the deployed site domain) and unlikely to
   change. Trade-off: if you ever migrate domains, edit the workflow
   file. Alternative: GitHub Actions Variables for `LEADS_UPLOAD_URL`
   so it's editable without touching YAML.
6. **Bot identity — `leads-cron[bot]`** for git author/committer.
   Consistent name in `git log`; visually distinguishes cron commits
   from human commits.
7. **Python 3.13** to match local dev. pyproject.toml allows 3.11+,
   but pinning to one version avoids "works on my machine" drift.
8. **Pip cache via `actions/setup-python@v5`** keyed on
   `pipeline/pyproject.toml`. Cuts cold-start time from ~90s to ~20s on
   warm runs.
9. **Timeout 30 min.** Real runs should be 5–10 min based on M8 smoke;
   30 min is safety margin if Gemini rate-limits cause many retries.
10. **No alerting/notifications hookup.** GitHub emails the repo owner
    on workflow failure by default — adequate for a personal project.
    Add Slack/PagerDuty later if needed.
11. **No `git pull --rebase` before push.** Race window (user pushes
    a commit while cron is running) is small for a personal project.
    If push fails with "non-fast-forward", the workflow fails, you
    notice in email, manually re-run. Adding rebase logic complicates
    the script for a rare event.
12. **No matrix / multi-region runs.** Single ubuntu-latest runner is
    plenty for the workload. ROADMAP explicitly out-of-scope.
13. **Commit granularity — single commit `add daily-leads cron
    workflow`.** Touches only `.github/workflows/daily-leads.yml` (and
    this plan doc).

## Implementation order

1. Create `.github/workflows/daily-leads.yml` with the YAML above.
2. Ensure all required secrets are set on GitHub (Settings → Secrets
   and variables → Actions). **You'll do this** since it requires
   your account.
3. Commit + push the workflow file.
4. **Manual smoke**: trigger via the Actions tab → "Daily lead refresh"
   → "Run workflow". Watch the run, confirm:
   - Pipeline step exits 0
   - Commit step pushes `leads.db` + `leads.json` back to master
   - Vercel Blob updated (verify via API: `curl
     https://www.ishaangpta.com/api/generate-leads?niche=mssp | jq
     .generated_at` returns a timestamp from within the run window)
5. Watch the next *scheduled* run (within 24 hours) to confirm:
   - It picks up the previously committed DB (not a fresh empty one)
   - `enrichment.enrich`'s skip-if-set logic kicks in for already-
     enriched leads (LLM call count is much lower than the first run)
   - DB only commits if there are actual new signals/scores

## Out of scope (per `pipeline/ROADMAP.md`)

- Multi-region / multi-runner setup.
- Retries on partial source failure — handled at app level via M7's
  per-source try/except.
- Workflow failure alerting beyond GitHub's default email.
- Backup of `leads.db` outside git (git history is the backup).
- Lock files / advisory locks for concurrent runs (concurrency block
  handles it within Actions; cross-runner concurrency isn't relevant
  with one runner).
- Cost monitoring / per-run LLM spend tracking. Add when bills get
  noticeable.
- Cron schedule per niche (e.g. breaches more often than jobs) —
  premature; one daily cron handles all sources fine.
